01 — What is a recipe

Not a prompt. Not a template. Not a packaged skill.

A recipe is a transferable, verifiable build intent — prose an agent reads once and follows to arrive at a capability you already have, on ground it has never stood on. The name comes from spore: the minimal portable unit that carries enough to grow the organism in a foreign environment. A recipe carries the same minimum — principles, contracts, scars — and leaves behind everything that only makes sense in your repo.

02 — How it works

Author once. Rebuild anywhere. Improve on every return.

The author side

  1. sporo harvest — mine your repo's own record; scars come from history, not memory
  2. sporo new — a coached draft, scar candidates pre-seeded
  3. sporo lint — the genre gate: shape, scars, neutrality
  4. sporo seal — version + content hash; a sealed recipe never silently mutates
  5. sporo export — one self-contained file. Hand over this, never the source

The reader side

  • Probe, don't assume — every precondition is checked against the real repository first
  • Map roles, never substitute — the recipe names the facts file, the collector; the reader decides where each lives here
  • Agree the outcome — which outputs, in what form, confirmed with a human before the first step
  • Conform — exact-bound schemas are checked mechanically, from the handed-over file alone

The loop

  • Report-backs are new scars
  • New scars raise the version
  • The recipe you read today is worth more than the one written a month ago — someone already paid down a failure you would otherwise hit
  • The moat isn't the file format. It's this loop.
author→ export→ rebuilt on a foreign stack→ report back→ new scar → new version↺
03 — Why prose, not packages

Readable before it runs. Self-checking after.

A skill package is opaque until it runs: you trust it or you audit its code. A recipe is prose — a human reads it, understands exactly what it will do, and decides whether to let an agent do it, before a single line executes. And the trust is structural, not promised: every build step carries a Done when: line, verification gates ship with the capability, and schemas another system consumes are exact-bound — checked mechanically in every adopter's CI. sporo doesn't tell you the result will be correct. It makes the result check itself.

04 — When not to use it

Recipes are not the right tool everywhere.

  • Same stack, same harness, same company — hand over the code, or package a skill. A recipe re-derives what you could have copied.
  • Determinism matters more than adaptability — zero re-derivation cost beats a document an agent has to interpret.
  • The capability is one library call — the recipe is the install command.
  • Nothing went wrong building it — a recipe's highest-value section is its scars. A build with no scars didn't need one.

Recipes earn their cost where the destination is genuinely heterogeneous: a different stack, a different agent, a different company — and code that literally cannot be handed over. That's the wedge: private teams transferring capabilities across a boundary code can't cross.

05 — Get sporo

One binary. No runtime. No dependencies.

$ curl -fsSL sporo.dev/install.sh | sh
$ sporo init          # install the authoring surface into a repo
$ sporo genre         # read the rules the gate enforces
Updating later: sporo upgrade (the binary), then sporo update in each repo (its skills). The binary hints — one quiet line, at most once a day — when a newer release exists.

macOS (Apple Silicon & Intel), Linux, Windows — amd64 and arm64, checksums published with every release.

Pre-release, source on GitHub. go install sporo.dev/sporo/cmd/sporo@latest resolves once sporo.dev is live; a brew tap follows.
